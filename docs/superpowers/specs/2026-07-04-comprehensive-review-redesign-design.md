# 校园失物招领系统 · 综合审查与 UI 重设计

- **日期**：2026-07-04
- **状态**：已批准，待实施
- **范围**：代码 Review & Debug、测试数据 Demo、UI 配色与视觉精修（暮山紫霭方案）

---

## 1. 背景与目标

校园失物招领系统当前已完成核心功能开发（19 Controller / 19 Service / 18 JSP / 13 表），前端使用 v6 经典蓝白配色（#3B82F6）+ layui + JSP。本次工作旨在：

1. **代码 Review & Debug**：全面扫描业务逻辑错误、遗漏功能、安全与权限问题、性能问题，并验证作业要求符合性。
2. **测试数据 Demo**：生成覆盖全部 13 张表的真实校园场景测试数据，便于报告演示。
3. **UI 配色与视觉精修**：将 v6 经典蓝白替换为"暮山紫霭"配色（雾紫 #7C6B9E），精修 Hero/KPI/卡片/图表视觉。

终极目标是让系统在功能完整性、数据丰富度、视觉美观度三方面同时达标，便于撰写实验报告与录制演示视频。

---

## 2. 整体架构与三阶段流程

采用**分层串行**执行方案，保证每一步建立在上一步正确基础上，避免返工。

```
阶段1: 代码 Review & Debug          阶段2: 测试数据 Demo              阶段3: UI 配色与视觉精修
┌─────────────────────────┐      ┌─────────────────────────┐      ┌─────────────────────────┐
│ • 19 个 Controller 全扫 │      │ • SQL 脚本 (demo-data.sql)│      │ • main.css 暮山紫霭配色  │
│ • 19 个 Service 全扫    │  →   │ • 13 张表全覆盖          │  →   │ • Hero/KPI/卡片视觉精修  │
│ • 作业要求符合性检查    │      │ • 真实校园场景           │      │ • 18 个 JSP 视觉统一     │
│ • 修复 + 验证编译       │      │ • 覆盖所有功能场景       │      │ • ECharts 配色更新       │
└─────────────────────────┘      └─────────────────────────┘      └─────────────────────────┘
        ↓                                  ↓                                  ↓
   Maven 编译通过                  数据可正常加载渲染                  视觉一致性验收
```

### 阶段间依赖

- 阶段2 测试数据依赖阶段1 修复后的业务逻辑（避免数据基于错误逻辑生成）。
- 阶段3 UI 精修依赖阶段2 的真实数据（验证数据渲染效果、图表配色）。

### 交付物

1. 本设计文档（`docs/superpowers/specs/2026-07-04-comprehensive-review-redesign-design.md`）
2. 阶段1：修复后的 Java 代码 + review 报告（嵌入实施计划）
3. 阶段2：`sql/demo-data.sql` 测试数据脚本
4. 阶段3：更新后的 `main.css` + 18 个 JSP 视觉精修

### 3.3 修复策略

- **P0（必须修）**：业务逻辑错误、安全漏洞、运行时 NPE、作业要求不达标项

## 6. 错误处理与测试策略

### 6.1 错误处理原则

| 阶段 | 错误处理 |
|---|---|
| 阶段1 代码修复 | 每修一处立即 `mvn compile`；遇编译错误回滚该处改动；保留 review 报告记录所有发现 |
| 阶段2 测试数据 | SQL 脚本幂等（先 DELETE 再 INSERT）；外键依赖按顺序插入；执行后 `SELECT COUNT` 验证每表行数 |
| 阶段3 UI 精修 | 改动前备份 `main.css`；每改一组 JSP 后浏览器刷新验证；保留 `v6` 配色注释便于回滚 |

### 6.2 测试策略

- **单元测试**：阶段1 修复后运行现有 6 个测试类，确保不破坏
- **集成验证**：阶段2 数据加载后，手动访问关键页面验证渲染
- **视觉验收**：阶段3 完成后，浏览器逐页截图对比

### 6.3 不做的事（YAGNI）

- 不引入新框架（Vue/React/Tailwind）
- 不重构现有 Controller/Service 分层结构
- 不增加新功能（仅修复 + 美化 + 数据）
- 不写新增的单元测试（除非修复的 bug 需要回归测试）
